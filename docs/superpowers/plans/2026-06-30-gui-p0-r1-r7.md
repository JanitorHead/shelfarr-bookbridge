# GUI P0 (R1–R7) Implementation Plan

> **For agentic workers:** Implement task-by-task, TDD, one commit per task. Steps use `- [ ]`.

**Goal:** Fix the six "the GUI shows nothing / I can't drive it" P0 problems from the 2026-06-30 GUI audit: make every sync observable, make "Sync now" non-blocking with live status, turn text fields into typed widgets, replace raw cron with a visual schedule, and make the Goodreads RSS-vs-cookie choice explicit.

**Architecture:** Server-rendered Go (`net/http` + `html/template` + `go:embed`). One CSS file + one tiny vendored `app.js` for progressive enhancement. Run observability is persisted at the single choke point `runOnce` (cmd/bookbridge/main.go) through which CLI sync, daemon cycles and the GUI button all flow.

**Tech Stack:** Go 1.26, `modernc.org/sqlite` (no CGO, **no WAL**), `github.com/robfig/cron/v3`, stdlib HTML templating. No SPA framework, no CDN.

## Global Constraints

- **Toolchain:** Go lives at `C:\Program Files\Go\bin` and is NOT on PATH. Prefix every Go command: PowerShell `$env:PATH = "C:\Program Files\Go\bin;" + $env:PATH`; bash `export PATH="/c/Program Files/Go/bin:$PATH"`.
- **Working dir:** the repo is `C:\source\repo\shelfarr-bookbridge`; the shell starts elsewhere — `cd` there first. After every change: `go build ./... && go vet ./... && go test ./...` must be green.
- **No WAL** in `store.Open` — never reintroduce it (Unraid `/mnt/user` FUSE crash). Default rollback journal only.
- **Image runs as root** (distroless `static-debian12:latest`, not `:nonroot`) — do not change the Dockerfile base.
- **Daemon/GUI must start WITHOUT Shelfarr config** — never make config loading fatal; gate runs with `cfg.ShelfarrConfigured()`.
- **Keep CSP strict:** `Content-Security-Policy: default-src 'self'` stays. NO inline `<script>` and NO inline `style=` attributes — everything via `/static/style.css` and `/static/app.js` (both same-origin, already allowed). This is the whole point of R1; do not "fix" rendering by loosening CSP.
- **Checkbox/select POST gotcha:** the Settings POST loop saves only non-empty values, so an unchecked checkbox submits nothing and could never turn a setting OFF. Checkbox fields MUST always write a canonical on/off value.
- **PowerShell hook:** commands containing `rm`/`Remove-Item` together with the string `C:\Program` are blocked. Don't delete files via shell during these tasks; use `git rm` only if a file must go, and avoid printing the PATH.
- **Commit per task** with a `Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>` trailer. **Do NOT push** — pushing triggers the image-publish CI; the orchestrator pushes once at the end after full verification.
